# Standard read-only review queries

Run these during loop step 7 (review), via read-only introspection (native MCP / equivalent) or by asking the author to report editor state. Each query has a *why* and a library route. Add 2–4 mechanic-specific queries derived from the written intent (each "the architecture should X" → one query proving or disproving X). Findings go out as triples (what → why it matters → where next + question); never fix anything.

1. **Call sites of every function the intent names.** Zero call sites = a dead wire: the function can exist, be correctly written — and do nothing. The canonical case: a profile-applying function with 0 call sites that shipped invisibly because demo data matched the defaults. → <https://martinsafka.github.io/gamedev-skripta/zapisky/dokumentace-jako-audit/>
2. **Write-only variables.** Filled but never read = a designed protection missing its last wire; the symptom (visual pop, wrong direction) lands far from the cause. Blueprint projects accumulate this debt exactly like C++ — grep just sees it worse. → <https://martinsafka.github.io/gamedev-skripta/zapisky/dokumentace-jako-audit/>
3. **Write-path tested with non-default values.** Any setter/config path must be exercised with values that *differ* from the class defaults — demo data equal to defaults masks dead wires. → <https://martinsafka.github.io/gamedev-skripta/zapisky/dokumentace-jako-audit/>
4. **Casts across system boundaries.** For every `Cast To` that crosses a boundary (character ↔ component ↔ ABP ↔ other systems): "who now has to know whom — and will that matter in six months?" Interface candidates. Casts inside fixed parent–child pairs or the project's own framework classes are legitimate — the check is for the *conscious decision*, not a ban. → <https://martinsafka.github.io/gamedev-skripta/praxe/komunikace-blueprintu/> · <https://martinsafka.github.io/gamedev-skripta/zapisky/pravidlo-70-30/>
5. **State as booleans vs. tags.** `bIsDoingX` sprawl vs. hierarchical gameplay tags; and tag lifecycle — cleared frame-exactly (Anim Notifies / explicit events), not via Delay nodes. → <https://martinsafka.github.io/gamedev-skripta/praxe/gameplay-tags/>
6. **Thresholds and tuning values: graph vs. data.** Numbers the intent calls "tunable" hardcoded in nodes → they belong in data assets / profiles (designer-editable, per-character swappable). → <https://martinsafka.github.io/gamedev-skripta/zapisky/pravidlo-70-30/> · <https://martinsafka.github.io/gamedev-skripta/praxe/principy-architektury/>
7. **One data contract vs. scattered reads.** Does consumer-facing data (typically the ABP) flow through one struct/contract copied thread-safely, or through scattered direct reads of the owner? Scattered reads couple layers and break the drop-in property. → <https://martinsafka.github.io/gamedev-skripta/zapisky/pravidlo-70-30/> · <https://martinsafka.github.io/gamedev-skripta/praxe/mm-zaklady/>
8. **Event Tick polling where events would do.** Per-tick checks for rare conditions → overlaps, dispatchers, notifies. → <https://martinsafka.github.io/gamedev-skripta/praxe/interakce-bez-event-ticku/> · <https://martinsafka.github.io/gamedev-skripta/praxe/komunikace-blueprintu/>
9. **Interface completeness.** Declared interfaces with unimplemented or unwired functions (declared ≠ delivered).
10. **Animation-state mechanics only: selection in data, entries pose-matched.** Animation selection should live in data (chooser rows / databases per state, swappable via tags), and state entries should not hard-start at time 0 when a pose-matched start is available. → <https://martinsafka.github.io/gamedev-skripta/praxe/gasp/> · <https://martinsafka.github.io/gamedev-skripta/zapisky/pravidlo-70-30/>
