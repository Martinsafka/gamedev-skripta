# Devlog jako mapa

**Kontext:** Pixin je můj 2D adventure engine s editorem — postavený za čtrnáct vývojových dní (léto 2026) v tandemu s AI agentem (Claude Code), publikovaný jako npm balíček a vzápětí otestovaný jam hrou, kterou na něm tým postavil za 48 hodin. Tenhle zápisek ale není o enginu. Je o devlogu, který při tom vznikl — 205 záznamů psaných automaticky po každém uzavřeném celku — a o tom, co se s takovou mapou dá dělat.

## Co se stalo

Nejcennější artefakt celého projektu je devlog sám, a vím to díky kontrolní skupině, jakou si nenaplánuješ: CLS_MM o dva měsíce dřív vznikla bez logu a rekonstrukce jejího vývoje byla týdenní archeologie (viz [dokumentace jako audit](dokumentace-jako-audit.md)). Pixin vznikal se zápisem po každém dokončeném celku — a výsledkem je kompletní mapa čtrnácti dní včetně slepých uliček, revizí rozhodnutí a přiznaných regresí. Stejný vývojář, srovnatelné tempo, dva měsíce po sobě: **„zapisovat čitelně" není režie, je to rozdíl mezi archeologií a mapou.** Bonus, který nikdo neplánoval: agent nemá paměť mezi sezeními, takže devlog je doslova jeho pamětí — disciplína zápisu se vynutila technickou nutností. (Formát What/Why/How/Follow-ups vydržel všech 205 záznamů.)

Druhá vrstva logu je **verifikační hranice**. Každý záznam končí výčtem, co je ověřené strojově — typecheck, lint, build, testy čisté logiky — a poctivým přiznáním, že „the render is eyeballed": vizuální pravda patřila člověku. Z té hranice plynula i designová rozhodnutí: ploché barevné výplně místo gradient API, s odůvodněním, že chyba v gradientu je runtime error neviditelný pro typecheck — volba API řízená ověřitelností, ne estetikou. Šestý den si agent **postavil oči** (Playwright skript nad reálným prohlížečem) a od té chvíle záznamy hlásí „verified in a real browser: 0 console errors". Hranice se neposunula rétorikou, ale nástrojem. A vedle toho drobný operační vzor: blokátor se řeší generováním (chybějící audio → syntetické placeholder WAVy) nebo zápisem fallback plánu — ne čekáním.

Tři ságy z logu ukazují, co mapa umožňuje:

- **Pathfinding se měří statisticky, ne anekdotou.** Po regresi (změna souřadného prostoru rozbila triangulaci — degenerované trojúhelníky, postava chodila „přes celou mapu a zpátky") nepřišla další záplata, ale přepis na visibility graph, robustní vůči tomu, jak se polygon rozřeže. A hlavně verifikace: randomizovaný test tisíců cest — před opravou 4,6 % cest s víc než 2,5× oklikou (nejhorší 51×), po ní nula cest mimo walkable a nejhorší oklika legitimní.
- **Po N. pokusu uvnitř krabice změň krabici.** Šestidílná sága ořezaných písmen v canvas textu: pět poctivě zalogovaných pokusů (měření plného textu, timing, padding, synchronní metriky, vlastní word-wrap), pak kapitulace s hlavou vztyčenou — bubliny do DOM vrstvy, která text umí nejlíp; engine jen publikuje pozice. Právě log s historií pokusů dovolí říct „N stačilo" s čistým svědomím.
- **Riziková fúze jako řada bezpečných kroků.** Migrace editoru ze statického preview nad živý běh hry proběhla jako strangler fig: plánovaná sekvence reverzibilních kroků za feature flagem, s invariantem „engine nikdy neimportuje editor" a předem ohlášeným bodem, odkud není návratu.

Poslední kapitola: **první cizí stroje.** Jam přinesl tři bugy — a všechny jedna třída: stav zapečený v čase buildu nebo module-loadu místo čtený za běhu (closure nad prázdným dokumentem, DEV konstanta knihovny vnucená konzumentovi, per-origin draft stínující každý nový projekt na localhostu). O pět dní později následoval systematický audit editoru s explicitním zdůvodněním: jam crashe „patřily do jedné třídy; tohle mapuje zbytek té třídy". Tři instance → hon na celou třídu. A detail na závěr: audit má vlastní errata — jeden bod byl po vstupu uživatele opraven. I audity se auditují.

## Co si z toho beru

- **Devlog není reporting, je to nástroj.** Statistická verifikace místo anekdot, licence na kapitulaci po N pokusech, mapování tříd chyb — nic z toho bez logu neexistuje, protože chybí historie, o kterou se to opírá.
- **Hranici „co umí spolupracovník ověřit" posouvej nástrojem, ne důvěrou.** A dokud se neposunula, respektuj ji v návrhu: ověřitelné API přebíjí hezčí API.
- **Bugy z prvního kontaktu se světem chodí ve třídách.** Fix instance je záplata; zmapování třídy je oprava.
- **Dogfooding ve dvou patrech:** demo hru na enginu stavěl sám agent přes authoring skills — demo testovalo engine a stavba dema testovala skills, obojí dřív, než přišel první cizí uživatel. (A z demo fáze vzor k zapamatování: stabilní event hook, jehož spouštěč se může volně měnit — zamkni rozhraní, odlož implementaci.)

> **Pozn.:** Jedna větev za samostatnou zmínku: protože všechna herní logika mluví jedním serializovatelným slovníkem (podmínky a efekty jako data), šlo čistou funkcí vygenerovat graf „kdo který flag zapisuje a kdo na něj reaguje" — dokument, který se umí sám zauditovat. To je nástrojová odpověď na třídu problémů „jedna veličina, dvě definice" z [GDD review](gdd-review.md).
